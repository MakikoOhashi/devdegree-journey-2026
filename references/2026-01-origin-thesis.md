# 2026-01 原点メモ（DevDegree不合格直後）

## テーマ
**「いらないもの」をシステムレベルで消す。**

## 当時の言語化（要約）
- 目標は機能追加ではなく、作業・確認・思考負荷の削減
- 「AIアプリを作る」より「やらなくていい仕組みを提供する」が本質
- 予測の価値は精度だけでなく、行動を合理化する根拠提示にある
- 収益は目的ではなく、副作用として発生する

## プロダクトへの対応
- TrackToInventory / Forecast: 「情報を取りに行く無駄」を消す
- WAKARUMADE: 「親子の説明摩擦・付き添い拘束」を減らす
- 税務支援構想: 「調べる・確認する・考える」時間を最小化する

## 原動力メモ（エモーショナル版）

### 各プロダクトの本当の目的

TrackToInventory / Forecast:
- 在庫管理アプリを作りたいのではない
- 「入荷を調べる」を消したい
- 「予測し続ける不安」を消したい
- 読めば分かる状態を先に作りたい
- つまり「毎回情報を取りに行く無駄」を消したい

WAKARUMADE:
- 教育アプリを作りたいのではない
- 「親が教え続ける負荷」を減らしたい
- 「親子の衝突」を減らしたい
- 「付き添い拘束」を減らしたい
- つまり「感情摩擦の無駄」を消したい

### 増やすより、減らす
- 世の中: 機能を増やす / ユーザーを増やす / 売上を増やす
- 自分: 作業を減らす / 手数を減らす / 思考負荷を減らす
- この逆方向こそ、自分の設計思想の強度

### 税務・申告で消したいもの
- サポートツールを増やしたいのではない
- 「調べる時間」「考える時間」「確認する往復」を消したい
- 自動計算 + 根拠表示で、最後は合理的に写経できる状態を作る

### CSVを取らせない発想
- 法的要件は守る
- でも移行作業は人間に押し付けない
- 出力・移行はシステム側で完結させる
- 「ユーザーの手を動かさない」を設計要件にする

### 本当の敵
- 敵は「人間が毎日繰り返している無駄な作業」
- 代行ではなく、仕組みで不要化する
- 「この作業、消えます」を実装で証明する

### 収益の見方（モチベーション）
- 収益は目的ではなく、消した無駄の評価額
- 例: 3社 x 月10万円 = 月30万円
- 意味: 「消された作業の価値」が金額として可視化された

### 面接での芯
- 「ビジネスを成長させたい」より
- 「EC業界の無駄な手作業と確認作業を消したい」
- Textile Trading で見た現実:
  - 「毎日毎日、誰かが手で入力してる」
  - 「毎日毎日、誰かが同じことを確認してる」
- TrackToInventoryで実証し、次はCSでより大きな構造に拡張する

### 自分への一文
**ミニマリスト思想 x エンジニア実装力 = いらないものをシステムレベルで消す人。**

## 現在の思想との接続
この原点は、現在の以下に接続されている。

- 不確実性の制御
- Forecast × 説明責任
- 人間とAIが一緒に責任を取るためのインフラ

# Agent Processors（Orchestrator / Processor 分離メモ）

Azure ハッカソン（Shelf）の設計メモ。
狙いは「単一AIが全部やる」ではなく、責務を分けて運用可能性（止まる/戻る/追跡する）を上げること。

## 1. 役割

### Orchestrator（統制・流れ）
- 入力を受け取る（メール/フォーム/Slack/手入力など）
- Processor を呼び出す（必要なものだけ）
- 結果を Issue にまとめる（Human in the loop の入口を作る）
- 状態遷移とログ出力を統制する

### Processor（個別タスク）
- 分類（カテゴリ/種類/優先度）
- 抽出（必要情報の抜き出し）
- 紐付け（過去事例/規程/顧客/案件/Shipment へのリンク）
- 提案（次のアクション案、確認質問の生成）

## 2. 状態（Issue / Workflow）

最小限の状態を固定する（増やしすぎない）。

- `new`: 入力受信直後（まだ整理されていない）
- `triaged`: Processor 結果がまとまり、論点が見える状態
- `pending clarification`: 情報不足で先へ進めない（保留キューへ）
- `needs human review`: 人間の判断待ち（承認センター）
- `approved`: 実行許可（次工程へ）
- `executing`: 実行中
- `done`: 完了
- `blocked`: 外部要因で停止（相手待ち/システム障害など）

## 3. `pending clarification queue` の意味

AI 業務システムは「情報不足で止まる」ケースが必ず発生するため、止まった案件を吸収する受け皿を正式に用意する。

- 目的: 失敗を隠すのではなく、運用上の“止まり方”を設計に含める
- 出力: 追加で必要な確認事項（質問リスト）を生成し、人間へ返す
- 効果: “AIが全部理解する前提”から、“不完全入力を運用で捌く前提”へ移行できる
